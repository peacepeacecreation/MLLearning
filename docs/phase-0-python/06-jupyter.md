# Урок 06 — Jupyter workflow: інтерактивний EDA

## Що це

**Jupyter Notebook** — інтерактивне середовище де ти пишеш код по клітинках і бачиш результат одразу під кожною. Файл `.ipynb` — це **JSON з клітинок**, кожна містить код, текст або збережений вивід.

Три складових Jupyter-системи:

- **Kernel** — фоновий Python-процес що виконує твій код. Тримає state (змінні, імпорти) доки живий.
- **UI клієнт** — те що ти бачиш і чим клікаєш: Jupyter Lab у браузері, VS Code, Cursor, PyCharm.
- **IPython/ZMQ протокол** між ними.

Ключова архітектурна річ: **kernel і UI — окремі речі**. Kernel — це Python-процес, UI — це редактор/веб-сторінка яка з ним спілкується. Тому `jupyter lab` і Cursor, і VS Code роблять одне і те саме — просто різні UI над одним kernel-процесом.

Аналогія що працює: **kernel = живий REPL у терміналі** (`python3`/`node`). Клітинки — це "команди" які ти надсилаєш у REPL. Файл `.ipynb` — це журнал того що надіслав + що REPL відповів.

## Для чого

Три сценарії де notebook кращий за `.py` скрипт:

1. **EDA (Exploratory Data Analysis).** Ти читаєш новий датасет — `df.head()`, дивишся, потім `df.groupby(...)`, будуєш графік, думаєш "цікаво, а що якщо...". Кожен експеримент — одна клітинка. Не треба перезапускати весь скрипт щоб змінити одну лінію. Це стандартний спосіб роботи в data science.
2. **Звіт-як-документ.** Чергуєш code-клітинки з markdown-клітинками. Виходить документ типу "аналіз → графік → інсайт → висновок". Експортуєш у HTML — маєш готовий звіт для клієнта/команди/портфоліо. GitHub рендерить `.ipynb` автоматично, тому це прямий портфоліо-артефакт.
3. **Прототипування ML.** Обучаєш модель, дивишся на метрики, підлаштовуєш параметр — ітеративно, без перезапуску всього тренування.

Що notebook **робить погано**:
- Production-код (батчева обробка, cron, API). Треба `.py` файли, тести, CI.
- Довгоживучий код в команді. Notebook важко diff-ити у git.
- Строгі reproducibility гарантії — стан kernel-а змінюється залежно від порядку запуску клітинок.

Правило: **prototype in notebook → productionize in `.py`**.

## Розбір

### 1. Запуск

Ти вже поставив `jupyter` через `uv add jupyter`. Два способи:

**A) Jupyter Lab у браузері:**
```bash
uv run jupyter lab
```
Відкриється браузер з файловим менеджером. Клік "New" → "Python 3" → створиться новий `.ipynb`.

**B) VS Code / Cursor:**
Просто створи `.ipynb` файл у редакторі. Extension автоматично спавнить kernel через `ipykernel_launcher`, підключиться через IPC, показуватиме клітинки у своєму UI. Треба один раз вибрати "Select Kernel" → твій `.venv`.

Обидва способи еквівалентні для kernel — різняться тільки у зручності UI.

### 2. Клітинки

Три типи:
- **Code cell** — Python-код. Виконується при `Shift+Enter`. Вивід (`print`, останній вираз, графіки) з'являється під клітинкою.
- **Markdown cell** — форматований текст. Заголовки `#`, списки, LaTeX через `$...$`. Використовується для структури звіту.
- **Raw cell** — плоский текст без обробки. Рідко потрібна.

Toggle між code/markdown — клавіші `M` (у markdown) / `Y` (у code), коли клітинка обрана але не редагується.

**Головне спостереження:** остання експресія у клітинці **автоматично друкується** без явного `print()`. Це відмінність від `.py` скрипта.

```python
# У клітинці:
titanic.head()   # → таблиця виводиться під клітинкою

# У .py скрипті потрібно:
print(titanic.head())
```

### 3. Kernel state — головна пастка

Це найважливіша концепція Jupyter. **Стан живе в kernel, не у файлі.**

Уяви часову лінію:

```
t=1  Cell 1 запущено: x = 10        → REPL state: x=10
t=2  Cell 2 запущено: x = 20; print → REPL state: x=20, вивід "20"
t=3  Cell 3 запущено: print(x)      → REPL state: x=20, вивід "20"
────
t=4  Ти редагуєш Cell 1 у файлі: x = 100 (Cell 2, 3 не чіпаєш)
t=5  Ти запускаєш ТІЛЬКИ Cell 1     → REPL state: x=100
t=6  Ти дивишся у файл — Cell 3 показує вивід "20" (застряг з t=3)
```

У момент t=6 notebook у **невідтворюваному стані**:
- Cell 1 у файлі каже `x = 100`.
- Cell 3 показує вивід `20`.
- Якщо хтось відкриє цей файл і запустить згори вниз — отримає `100`, не `20`.

**Це і є проблема reproducibility.** Те що бачиш ти ≠ те що побачить інший читач.

### 4. Restart & Run All як reproducibility-тест

Правило:

**Перед комітом у git — завжди `Kernel → Restart Kernel and Run All Cells`.**

Що це робить:
1. Вбиває поточний kernel — весь state очищується.
2. Стартує новий kernel.
3. Запускає клітинки згори вниз, кожну по черзі.
4. Якщо якась клітинка залежала від "прихованого" стану (змінна створена в клітинці яку ти видалив, наприклад) — тут вона впаде.

Після цього номери клітинок мають бути `[1] → [2] → ... → [N]` строго послідовно. Це візуальний індикатор що notebook пробіжить у fresh clone.

Якщо номери типу `[3], [1], [7], [2]` — це червоний прапорець. Ти запускав клітинки у довільному порядку. У тебе працює, у мене — можливо ні.

### 5. matplotlib у notebook — переваги

**`plt.show()` не потрібен** — matplotlib автоматично рендерить figure в кінці клітинки. Це значить що проблема "накладання графіків" з Уроку 05 (де треба було розставляти `plt.show()` між викликами) **зникає**.

```python
sns.histplot(data=titanic, x="age", bins=30)
# ← кінець клітинки → графік з'являється автоматично
```

Кожна клітинка = свій figure. Без ручного управління.

**Косметика:** seaborn-функції повертають Axes object, notebook друкує його repr перед графіком:
```
<Axes: xlabel='age', ylabel='Count'>
```

Виправляється **крапкою з комою** в кінці рядка:
```python
sns.histplot(data=titanic, x="age", bins=30);
```

`;` придушує repr останнього виразу. У notebook це стандартний трюк.

### 6. Notebook як звіт — структура

Notebook стає **портфоліо-артефактом** тоді, коли ти чергуєш markdown-заголовки з code-клітинками. Читач бачить не купу коду, а історію: контекст → аналіз → результат → інсайт.

Стандартна структура для EDA-звіту:

```markdown
# Заголовок аналізу
**Мета:** одне речення про що і навіщо.

## 1. Огляд датасету
Розмір, типи колонок.
[code: df.info()]

## 2. Пропуски
Що чисте, що треба обробити.
[code: df.isna().sum().sort_values(ascending=False)]

## 3. Baseline метрика
Що модель мала б перебити.
[code: df["target"].mean()]

## 4. Виживання за фічею X
Гіпотеза + перевірка.
[code: groupby + barplot]

## 5-N. Інші зрізи...

## Висновки
- Bullet-list з ключовими інсайтами.
- Що з цього важливо для наступних кроків.
```

Ти зробив саме це для Titanic — 7 секцій + висновки, 9 клітинок разом. Це і є перший **портфоліо-артефакт**.

### 7. Експорт

Для показу нетехнічним читачам або як HTML на GitHub Pages:

```bash
uv run jupyter nbconvert --to html notebooks/titanic_eda.ipynb
```

Генерує самодостатній HTML з code, вивідом і графіками.

Інші формати: `--to pdf` (потребує LaTeX), `--to markdown` (для GitHub README), `--to script` (перетворює `.ipynb` → `.py` без output — корисно для code review).

### 8. Magic commands

Спеціальні команди що починаються з `%` (лінія) або `%%` (клітинка).

Найкорисніші:

```python
%matplotlib inline
# Графіки в notebook, не в окреме вікно (default у сучасних версіях)

%%time
# Міряє час виконання цілої клітинки
some_slow_function()

%who        # Список змінних у kernel
%whos       # Той самий список з типами і розмірами

%%bash
ls -la
# Виконує bash-команди прямо з клітинки

%load_ext autoreload
%autoreload 2
# При редагуванні .py-модуля, notebook перезавантажує його автоматично
```

## Gotchas

- **`.ipynb` порожній (0 байт) → `NotJSONError`.** Створювати тільки через Jupyter UI, не через `touch` чи "New File" у редакторі. Файл має валідну JSON-структуру що Jupyter генерує сам.
- **Kernel state не очищується коли ти видаляєш клітинку.** Змінні лишаються. Тому `x` може існувати в kernel навіть якщо у файлі клітинки `x = 10` вже немає.
- **Порядок виконання ≠ порядок у файлі.** Клітинки можна запускати як завгодно. Тому `Restart & Run All` перед комітом — обов'язковий.
- **`plt.show()` не потрібен у notebook**, але **потрібен у `.py`**. Не переноси notebook-код в скрипт сліпо.
- **`.ipynb_checkpoints/`** — папка з автобекапами. Має бути в `.gitignore` (у нас вже).
- **Комітити `.ipynb` з великими embedded output-ами** роздуває репо і робить diff-и нечитабельними. Опція: `nbstripout` pre-commit хук що чистить output-и перед комітом (буде у Уроці 07).
- **matplotlib `plt.title(...)` copy-paste між клітинками.** Легко забути оновити заголовок. `Restart & Run All` не ловить логічні copy-paste помилки, тільки reproducibility. Читай очима перед комітом.
- **Cell number `[N]` тільки показує коли останній раз запускалась**, не чи вона працює. Клітинка може мати номер `[5]` але містити застарілий вивід від `[3]` (не була re-run після зміни).
